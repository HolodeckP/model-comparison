What the coding task is

The model is given an existing program that processes incoming records (think of a form or order that arrives, gets sorted, and updates a stored list). Right now it has a bug, and the model has to fix it without breaking what already works. The job has four connected parts:

  1. Recognize a repeat submission. When an updated version of something already in the system comes in, the program currently mistakes it for a brand-new, unrecognized item. The model needs to add a check that spots "this is an update to an existing record" before the existing sorting happens.
  2. Compare old vs. new and reconcile. For an update, match each incoming entry to what's already stored: keep and update the ones that match, add the genuinely new ones, and mark the ones that disappeared as no longer active.
  3. Fix the labeling bug. Entries that are actually new are being mislabeled as "inactive" instead of "new." The model has to find and correct that.
  4. Hide the inactive entries downstream. The part of the program that produces the final document/output should leave out anything marked inactive, so it doesn't show up where it shouldn't.

Why it's a good way to compare models: it isn't a one-line fix. The model has to read unfamiliar code, understand how the existing decision-making flows, find where the logic goes wrong, and make several coordinated changes across different parts of the program — all while keeping the parts that already work intact. Cheaper or weaker models can often patch a single spot but struggle to hold the whole flow in their head and keep every change consistent.
